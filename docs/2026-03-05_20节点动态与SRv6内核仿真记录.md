# 20节点动态拓扑与SRv6内核仿真记录（2026-03-05）

## 1. 背景与目标

本文档记录 2026-03-05 在宿主机单机环境完成的三部分工作：

1. 20节点 OSPF 动态扰动全量回归（QoS、通断、时延、单播、广播、组播）。
2. 20节点动态拓扑性能基准（多轮链路扰动，产出 P50/P95/P99）。
3. SRv6 内核协议 canary 验证（从控制逻辑模拟推进到 Linux 内核 seg6/seg6local 实测）。

目标是验证：

- 当前 20 节点环境是否可稳定支持“业务运行中动态变更”。
- 拓扑动态扰动的关键时延指标。
- 后续 SRv6 仿真主线可行性（事件触发临时改路 + 内核协议链路可用）。

## 7. SRv6 内核协议仿真（最终打通）

### 7.1 脚本

- `scripts/test_srv6_kernel_canary_20.sh`

### 7.2 最终有效日志

- `logs/test_srv6_kernel_canary_20_20260305_133503.log`

### 7.3 验证目标

在 3 个节点（src/mid/dst）上验证 Linux 内核 SRv6：

1. src 使用 `ip -6 route ... encap seg6 ...` 注入 SRH。
2. mid 使用 `seg6local End.DX6` 执行 decap + 转发。
3. dst 正常收到业务 IPv6 报文并回包。

### 7.4 关键证据

- 基线（不加 SRv6）
  - `baseline_mid_icmp6_count=0`（流量未经过 mid）
- 加 SRv6 后
  - `ping6 0% loss`
  - mid 入口抓到 `srcrt`（SRH）
  - mid->dst 口抓到 decap 后 ICMPv6 请求
  - dst 入口抓到 ICMPv6 请求

### 7.5 本次关键修复点（很重要）

1. `veth_0.<vid>` 子接口的 `seg6_enabled` 不能用普通 `sysctl` 路径可靠设置。
2. 需通过 `/proc/sys/net/ipv6/conf/<iface>/seg6_enabled` 显式写入 `1`。
3. mid 的 SID 需为 `local SID + seg6local action End.DX6`。
4. 源端 SRv6 路由需带可达下一跳（`via`）并绑定正确接口。

## 9. 回滚与清理建议

若只回到纯 OSPF 基线：

1. 删除源端 SRv6 路由（目标前缀的 `encap seg6` 条目）。
2. 删除 mid 端 `seg6local` 本地 SID 路由。 
3. 保留 `lab_er_vlan_subif.sh check` 直到 `bad_neighbor_nodes=0, miss_total=0`。

如需彻底清场：

- `LAB_ID=erv20 LAB_NODES=20 scripts/lab_er_vlan_subif.sh down`
- 再 `up` 重建。

## 10. 后续建议（面向300节点）

1. 每分钟动态变更采用“增量+分批”，避免全网同时重算。
2. 验收口径分层：
   - 业务恢复：`T_recover_up` / `T_routes_ok_down`
   - 控制一致性：`T_full_converge`
3. SRv6 正式化方向：
   - 故障事件发布（上游检测）
   - 源端策略下发（临时 SR Policy）
   - 后台 OSPF 收敛后回切
4. 扩容到 300 前先跑 20->50->100 阶梯压测，固定同一指标口径。
