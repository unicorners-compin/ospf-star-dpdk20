# 主机内中心仿真器规范（冻结版）

## 1. 目标
构建“宿主机 FRR 容器 + 宿主机 OVS 引流 + 单 Docker 中心仿真器”的可验证环境，并保持后续向 SRv6 仿真演进。

## 2. 冻结范围（不允许偏移）
1. 本阶段不依赖 DPU，不依赖外部交换机配置。
2. 中心仿真器采用单 Docker、双 veth 口模型：
   - `sim_in`: ingress（OVS -> simulator）
   - `sim_out`: egress（simulator -> OVS）
3. 中心仿真器按规则执行：
   - `forward/drop`
   - 可选损伤：`delay/loss/rate`
4. OVS 仅负责引流和回流分发，不承载策略。
5. 先固定 `3+1`（3 节点 FRR + 1 仿真器），验证后扩到 `20+1`。
6. 策略模型必须预留 SRv6 字段（locator、sid/policy），即使本阶段不启用。

## 3. 明确不做（本阶段）
1. 不改交换机配置。
2. 不接入 DPU 数据面。
3. 不做动态拓扑切换（先静态图验证）。
4. 不做复杂控制面数据库（先 JSON 规则文件）。

## 4. 数据平面路径
1. FRR 容器 -> 宿主机 OVS -> `sim_in`
2. 仿真器规则匹配：
   - 命中 `forward`：从 `sim_out` 发出
   - 命中 `drop`：丢弃
   - 未命中：默认丢弃（fail-closed）
3. `sim_out` -> 宿主机 OVS -> 按目标 MAC/组播广播 fanout 到 FRR 容器

### 4.1 网络互联细节
1. 本阶段完全在宿主机完成：OVS bridge + veth peer + docker netns。
2. 中心仿真器容器仅连接两根 veth，不加入容器网桥。
3. 后续若引入 DPU 或外部交换机，必须先更新本规范并重新审查。

## 5. 安全与防环要求
1. 默认全丢弃，显式放行。
2. 宿主机OVS先下保护流（drop-all），检查通过后再切业务流。
3. 任一异常触发自动回滚到drop-all：
   - OSPF邻居异常抖动：60秒内出现>=3次 `Full -> 非Full`
   - 软中断异常升高：任一核心 `ksoftirqd` 持续>85% 且持续>=15秒
   - 日志环路/风暴特征：30秒内命中关键字（`own address as source` 或 `net_ratelimit`）>=20次

## 6. 配置接口（规则）
规则文件：`sim_policy_*.json`
最小字段：
- `src_mac`
- `dst_mac`
- `action` (`forward` | `drop`)
- 可选 `pkt_type` (`unicast|ospf_multicast|arp_broadcast|broadcast|multicast`)
- 可选 `impair`:
  - `delay_ms`
  - `loss_pct`
  - `rate_mbps`

SRv6 预留字段（本阶段不启用但必须保留）：
- `srv6.enabled`
- `srv6.locator`
- `srv6.policies[]`（SID 列表、匹配条件、动作）

热更新机制：原子替换文件 + 进程 `SIGHUP` 重载。

## 7. 分阶段实施
### 阶段A：3+1 Canary
1. 拉起 3 个 FRR 容器 + 1 个中心仿真器容器。
2. 验证OSPF邻接、路由可达、规则命中。
3. 连续运行30分钟观察稳定性。

### 7.1 监控与可观测性（必选）
1. 中心仿真器需暴露以下指标：
   - `sim_in/sim_out` 收发包/字节计数
   - 丢包计数（按原因分类）
   - 规则命中计数
   - 进程CPU与内存使用
2. 宿主机需采集：
   - OSPF邻居状态变化计数
   - OVS端口丢包计数
   - `ksoftirqd` 核心占用
3. 指标需可导出到日志或接口，支持自动判定A/B阶段是否通过。

### 阶段B：19+1
1. 复用同一架构扩展到 20 节点。
2. 验证全量邻接与路由收敛。
3. 再逐步加入 loss/delay/qos。

## 8. 验收标准
### A阶段（3+1）通过条件
1. 30分钟内无非预期OSPF flap（邻居从Full掉线）。
2. 无环路/风暴类内核告警。
3. 规则 `forward/drop` 行为与预期一致。
4. 宿主机无明显卡顿，控制命令可及时响应。

### B阶段（19+1）通过条件
1. 邻接与路由表完整。
2. 无持续异常丢包。
3. 回滚机制可用（手动触发可立即恢复drop-all）。

## 9. 回滚策略
1. 组件与职责：
   - `host-watchdog`：检测触发条件，发出回滚指令。
   - `ovs-guard`：执行宿主机OVS切换到drop-all。
   - `sim-agent`：中心仿真器切到全丢弃模式。
2. 回滚动作顺序：
   - 宿主机OVS立即切回drop-all。
   - 中心仿真器切到drop-all（或停止转发）。
   - 保留现场日志与计数器快照。

## 10. 执行纪律
1. 本文档冻结后，任何架构变更必须先更新`spec.md`并再次审查。
2. 实施过程不得偏离第2节冻结范围。
3. 如发现偏离风险，必须先暂停执行并确认。

## 10.1 算法一致性硬约束（新增，强制）
1. 自本条生效起，SNN-SRA相关实现必须与用户提供仓库逻辑保持一致，不允许任何算法简化、替换或降级实现。
2. 明确禁止：以“RIP-like近似”“周期广播近似”“局部启发式近似”等方式替代仓库定义的SNN-SRA核心机制。
3. 明确禁止：基于简化算法产出的任何性能结论、对比结论、验收结论。
4. 若因工程限制暂时无法1:1实现，必须先停止测试与结论输出，先在`spec.md`提出偏差说明并获得审查同意后方可继续。
5. 任何提交都必须附带“与仓库一致性说明”；无法证明一致性的变更一律视为不合规。

## 11. 规则热更新约束
1. 禁止通过进程重启实现规则更新。
2. `SIGHUP` 仅允许触发“内存规则表原子切换”，不得重启数据面主循环。
3. 规则更新时间窗口内不得出现可观测转发中断（以连续探测流验证）。

## 12. 回滚触发验收（新增）
1. 人工注入三类触发条件（OSPF flap、软中断阈值、日志风暴关键字）。
2. 每类触发都必须在阈值命中后自动执行回滚，并在60秒内恢复可控状态。
3. 任一触发场景回滚失败，阶段验收直接不通过。

## 13. SRv6内核仿真补充（2026-03-05）
1. 本阶段允许在宿主机容器内做最小 SRv6 内核 canary（不改交换机）。
2. canary 验证口径：
   - 源节点存在 `encap seg6` 路由；
   - 中间节点存在 `seg6local` 本地 SID 动作；
   - 中间节点可抓到 SRH（`srcrt`）；
   - 中间节点到目的节点可抓到 decap 后业务包；
   - 端到端业务探测成功（如 `ping6`）。
3. 子接口注意事项：
   - 对 `veth_0.<vid>` 这类接口，`seg6_enabled` 可能无法通过普通 `sysctl` 路径可靠设置；
   - 允许通过 `/proc/sys/net/ipv6/conf/<iface>/seg6_enabled` 显式写入开启。
4. canary 只验证协议与链路可行性，不等价于生产级 SR Policy 控制器。

## 14. DPU安全开发红线（2026-03-06，强制）
1. 本节为最高优先级约束；与其他条款冲突时，以本节为准。
2. 未通过“预检+回退脚本审查”前，禁止执行任何DPU破坏性操作。

### 14.1 禁止项（绝对禁止）
1. 禁止在未100%确认管理链路归属前，对疑似管理口执行`dpdk-devbind`绑定到`vfio-pci`。
2. 禁止一次性大规模分配hugepage；禁止“先大配再验证”。
3. 禁止未写回退脚本就执行网卡驱动切换、hugepage调整、DPDK原生PMD切换。
4. 禁止在同一变更窗口内同时做“驱动切换 + hugepage调整 + 程序升级”三项变更。
5. 禁止在生产实验桥（如`erv300_br`）做未审查验证，必须先在独立小拓扑验证。

### 14.2 变更前预检（必须全部通过）
1. 输出并留档：`ip -br link`、`dpdk-devbind.py --status`、`/proc/meminfo | grep Huge`。
2. 明确管理口与数据口映射（接口名、PCI地址、用途）并写入本次变更单。
3. 校验可回退命令可执行（仅检查，不实际切换业务）。
4. 评估内存预算：必须保证OS保底可用内存>=2GB。

### 14.3 执行流程（单步、可验证、可回退）
1. 第一步只调整hugepage（小步），验证SSH稳定与系统可用。
2. 第二步只切换目标数据口驱动，验证管理口连通不受影响。
3. 第三步只启动DPDK程序，验证端口识别与最小收发。
4. 每一步失败立即执行对应回退，不允许继续下一步。

### 14.4 标准回退动作（模板）
1. hugepage回退：
   - `echo 0 > /sys/kernel/mm/hugepages/hugepages-524288kB/nr_hugepages`
   - `echo 0 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages`
2. 网卡驱动回退（示例）：
   - `echo <PCI> > /sys/bus/pci/drivers/vfio-pci/unbind`
   - `echo <PCI> > /sys/bus/pci/drivers/octeontx2-nicpf/bind`
   - `ip link set <iface> up`
3. DPDK进程回退：停止进程并恢复内核网络栈连通验证。

### 14.5 审批与执行纪律（强制）
1. 任何涉及`y/p/n`选择的高风险步骤，必须先征询顾问意见，再执行。
2. 执行者必须在每条命令前给出“目的、预期结果、回退命令”。
3. 未经明确同意，禁止修改交换机配置、禁止触碰非目标环境。
4. 发生异常时立即冻结变更，先恢复管理可达，再讨论功能推进。

## 15. 开发安全保障清单（顾问确认版，2026-03-09，强制）
1. 本节适用于宿主机(192.168.0.2)与DPU52(192.168.0.52)联调及DPDK开发全流程。
2. 本节与其他条款冲突时，以更严格条款执行。

### 15.1 强制门禁（每次canary/full前必须满足）
1. 必须先执行`precheck`并全通过。
2. 必须确认100Gbps数据面链路：
   - 宿主机数据口`ens31f1np1`速率=100000；
   - DPU数据口`eth0`速率=100000；
   - DPU管理口`eth2`仅用于SSH（可为1000）。
3. 必须完成本次变更代码/脚本审查并留档。
4. 必须在执行前明确通告本次变更目标与影响范围。

### 15.2 禁止项（绝不允许）
1. 严禁将管理流量跑到100Gbps数据面。
2. 严禁在DPDK应用运行时执行网卡驱动绑定/解绑。
3. 严禁修改交换机配置。
4. 严禁在线修改SSH/防火墙核心配置作为联调手段。

### 15.3 执行分级与审批
1. 读操作：日志查看、状态采集、`precheck`，可自审执行。
2. 写操作：部署二进制/脚本、`canary`，需审批后执行。
3. 高风险操作：固件/系统升级、压力满载测试、驱动切换，需维护窗口与双人审批。

### 15.4 故障应急SOP
1. SSH断连：
   - 先本机与宿主机连通性排查；
   - 再做DPU可达性确认；
   - 仍失败则升级到带外处理。
2. 链路down：
   - 优先采集`ethtool`与应用日志；
   - 仅在确认需要时重启DPDK进程；
   - 禁止同时改配置与重启多组件。
3. 进程卡死：
   - 先`SIGTERM`，超时再`SIGKILL`；
   - 清理测试残留（进程、netns/veth）；
   - 回滚到上一个稳定版本后再恢复验证。

### 15.5 最小化回滚策略
1. 优先关闭功能开关（如`BUM_FANOUT_ENABLE=0`）恢复旧路径。
2. 回滚到上一个稳定二进制与规则配置，不引入新变量。
3. 回滚完成后必须立即执行`canary`验证恢复。

## 16. Event模式试点（可回退强约束）
1. 目标：在不影响当前可用`poll`路径的前提下，引入`event`模式最小骨架验证。
2. 原则：任何时刻都可“一条命令”回退到`poll`基线。

### 16.1 试点范围
1. 仅限DPU52上的环境仿真器进程（`dpu-dualport-l2`）。
2. 不修改交换机配置，不修改300节点业务拓扑，仅切换仿真器进程模式。

### 16.2 执行步骤（强制顺序）
1. 先快照：`scripts/dpu_event_mode_guard.sh snapshot`
2. 再启动event：`scripts/dpu_event_mode_guard.sh start-event`
3. 执行回归：`scripts/server_dpu_safe_link_up.sh canary` -> `full`
4. 若任一步失败：立即执行`scripts/dpu_event_mode_guard.sh rollback`

### 16.3 判定标准
1. 功能判定：`full`必须`PASS=13 FAIL=0`。
2. 稳定判定：至少连续5轮`full`通过，期间无SSH异常、无进程异常退出。
3. 若event不可用或不稳定，允许自动回退到poll；不得阻塞当前业务验证。
